## アイデア① Gemini版を作る（AI部分だけ差し替え）

**作成日：2026-06-30**

### 何をしたいか
今のアプリのAI評価（週次コメント・月次コメント）部分だけを、Claude APIからGemini APIに差し替えた「Gemini版」を作る。本体ロジック（残高モデルα・集計）は一切触らない。

### なぜやりたいか（動機）
- **課金が気になって気軽にテストできない**のが本音の動機。1回が小さな金額でも、開発中に何度も叩くと「お金が減る音を聞きながら実験してる」心理的ブレーキがかかる。
- Geminiには**無料枠**があり、Flash系の軽いモデルなら家族ひとり運用＋開発テスト程度なら無料枠に収まる可能性が高い。「失敗してもタダ」なら気軽に試せる。

### なぜ今なら難しくないか
- 54番の「AI評価の観点刷新」で、**アプリ＝計算担当／AI＝コメント担当**に役割を分離済み。
- AIに渡すのは「計算済みの数字＋短いプロンプト」だけ。求めているのは100文字（週次）・500文字（月次）のコメント生成のみ。
- 難しい計算をAIにさせていないので、モデルの差が出にくい部分＝Geminiでも十分こなせる。

### 技術的に変える3か所
1. **呼び出し先URL**：Anthropicのエンドポイント → Geminiの `generateContent` エンドポイント。
2. **リクエスト/レスポンスの形**：messages配列の組み立て、APIキーの渡し方、レスポンスの取り出し（`data.content[0].text` に相当する箇所）がGeminiは構造が違うので書き換える。
3. **プロンプト本文**：ほぼそのまま流用可。「数字はアプリ算出済み・計算不要・100文字以内で一言」等の指示はGeminiでも素直に効く。

→ 実質「APIを叩く関数を1〜2個ラップし直す」イメージ。本体は無改修。

### 注意点
- **APIキー**：本アプリはキーをデバイス保存しているので「GitHub Pagesで誰でも見える」問題はクリア済み。
- **コメントの“味”が変わる**：同じプロンプトでもモデルごとに文体・褒め方・指摘のクセが違う。精度ではなく好みの問題。
- **無料枠の上限は時期で変わる**：本気で検討するときに、その時点の無料枠（回数・対象モデル）を確認すること。

### 進め方の案
いきなり本番化せず、「Gemini版」を別ファイルで並行運用 → しばらく両方のコメントを見比べ → 気に入った方を残す。

## アイデア② 「AIコメント：オン/オフ」スイッチ（採用したい）

**作成日：2026-06-30**

（旧称：テストモード。Gemini化するかどうかに関わらず、**この仕組み自体を今後採用したい**。）

### 何をしたいか
AI呼び出しをオフにできるスイッチを設定画面に置く。オフのときはAPIを一切叩かない（＝課金されない）。

### なぜ有効か
- 開発中、UIやメール組み立て・レイアウト確認など「コメントの中身は関係ない」テストが全部タダになる。
- 本物のコメントの質を見たいときだけオンにして実APIを叩けばいい。
- 家族にとっても「今日はAIいらない／計算だけしたい」というときに気軽に使える機能になる。

### 重要：計算は常に正しく出る
- 計算（残高モデルα・集計・固定費の実費/予算・おこづかい残高など）は**ぜんぶアプリ側が担当**。
- スイッチがやるのは「AI呼び出しをオフにする」だけ。**オン/オフに関係なく計算は常に正しい。**
- 変わるのはコメント文だけ。

### 決まった仕様（雑談で確定）
- **置き場所**：設定画面。家族に見えてOK（むしろ機能として使える）。AIボタンのそばでも可。
- **名前**：開発者向けの「テストモード」ではなく「**AIコメント：オン/オフ**」のように家族にも分かる表現。
- **オフのときの表示**：コメント欄に「**AIコメントはオフです**」と表示する。
  - （コメント欄ごと消すと「壊れた?」と不安になるため。一言出ていれば意図的なオフだと一目でわかる。）
- **オフのとき**：APIを一切叩かない＝課金ゼロ。

### 実装の考え方（技術ヒント）
- 週次（`runWeeklyCheck` / `shareWeeklyMail`）と月次（`generateMonthlyAI`）の**AI呼び出しを共通関数に通す**（例：`callAI()`）。
- スイッチの状態は `localStorage`（キープレフィックス `kr_`）に保存。例：`kr_ai_comment` のような形。
- 共通関数の先頭で「オフなら実APIに行かず即リターン」する分岐を置く。

\`\`\`javascript
// 例：AI呼び出しの共通入り口
async function callAI(prompt, kind = 'weekly') {
  const aiOn = localStorage.getItem('kr_ai_comment') !== 'off'; // 既定ON
  if (!aiOn) {
    return null; // 呼び出し側で「AIコメントはオフです」表示に分岐
  }
  // ↓ 本番はこれまで通り実API（将来Gemini化するならこの中身だけ差し替え）
  const res = await fetch('https://api.anthropic.com/...', { /* 既存のまま */ });
  const data = await res.json();
  return data.content[0].text;
}
\`\`\`

- **ポイント**：週次も月次も必ずこの共通関数を経由させる → スイッチ1個で全AI呼び出しが同時に切り替わる。
- **将来のGemini化との相乗効果**：差し替えるのは \`callAI()\` の中身だけで済む（アイデア①の「ラップし直すだけ」と同じ場所）。

### 運用上の注意
- スイッチは \`localStorage\` なのでデバイスごとに独立。
- コード側に \`true\`/\`false\` を直書きしたまま出す事故だけは防ぐ。**デプロイ前チェックに「AIコメントの状態を確認」を1項目足す**と安心。

## 📌 ファイル運用ルール（Claude向け・このファイルの扱い方）

> ※ここから下はファイル作成・更新時の作法メモです。

### このファイルの性質
- このファイルは**番号を振らない恒久ドキュメント**。新しいアイデアが出たら、このファイルに追記・更新していく。
- 確定タスクではなく「いつかやるかもしれない構想＋技術ヒント」を貯めておく場所。

### 命名ルール
- **★始まりのファイル＝恒久ドキュメント。** 中身を更新しても同じファイル名（`★構想メモ_やってみたいことノート.md`）を使い続ける。日付はファイル名に付けない。
- ファイル一覧をソートすると ★ が数字より前に来るので、先頭にまとまって埋もれない。
- （通常ルールの例外：コード・設計書などは従来どおり「通し番号＋日時」で新規出力。このファイルだけが例外。）

### 追記の作法
- 新しいアイデアが出たら「## アイデア③ …」として追記し、**冒頭の目次にも1行足す**（No.・アイデア名・作成日・ひとこと）。
- 各アイデアの見出し直下に **作成日** を入れる（日付管理はファイル名ではなくアイデア単位で行う）。
- アイデアの末尾に \`[▲ 目次へ戻る](#-目次アイデア一覧)\` を付けておくと往復しやすい。
- 確定して実装フェーズに移ったものは、印をつけるか実装フェーズ表（v8系）側へ移す。
- 先頭の最終更新日を更新する。
